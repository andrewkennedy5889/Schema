## 2026-04-24 — Auth/tenancy Phase J: admin queue authorization via RLS, not in-code filtering

**Affects**: `"01_tenancy"."org_member_requests"` RLS policies (`omr_select`, `omr_insert`, `omr_update`, `omr_delete`), `app/admin/queue/page.tsx`, `app/api/decide-member-request/route.ts`, `lib/email.ts`, migration `supabase/migrations/20260424000011_phase_j_org_member_requests_rls.sql` on `Nex4.23.26` branch `auth-tenancy-v2`, AUTH-TENANCY-PRD v2 §9 Phase J.
**Decision**: Shipped Phase J as four commits. (1) **RLS migration** idempotently drops + recreates the Phase A policies on `org_member_requests` so that authorization for the admin queue is enforced at the row level for every role path: SELECT visibility = `person_id = current_person_id() OR is_org_admin(org_id) OR is_system_admin()` (one policy covers requester + org admin + sysadmin; the sysadmin branch also provides the "fallback queue for admin-less orgs" called out in the PRD — no separate policy needed since `is_system_admin()` returns true regardless of whether any `org_admins` row exists for the target org); INSERT = `person_id = current_person_id() AND status='pending'` (real inserts come from the Phase F trigger under `SECURITY DEFINER`, this policy guards against client-side direct insert); UPDATE USING = sysadmin OR org_admin OR self; UPDATE WITH CHECK = sysadmin OR org_admin OR (self AND status='withdrawn'); DELETE = sysadmin only (approved/denied rows are audit history, no UI path exposes DELETE). Whatever the RLS policy returns is what the admin sees — there is zero org-filter logic in the page. Side-queries fetch Person names (`first_name`, `last_name`, `user_id`) and Org names/slugs (`name`, `slug`) by batched `.in()` calls, stitched client-side because `"Person's_org"` doesn't play nicely with PostgREST embed syntax on the quoted `Person(s)` table. The UPDATE carries an `.eq("status","pending")` guard plus a `.select("id").maybeSingle()` so a racing approval from another admin returns `already_decided` cleanly. Email lookup uses the service-role client (`lib/supabase/service.ts`) since `auth.users` is not RLS-readable by authenticated roles. (8) **Full E2E with real admin session** requires Supabase session cookies that the MCP tool can't mint — marked manual per PRD, covered by AUTH-T069/T071/T072/T073 which remain manual until Phase O. (10) **Security advisors post-migration**: no new findings; pre-existing INFO/WARN entries (`08_moments.moment_node(s)_*` RLS-no-policy, `public.touch_updated_at` + `public.enforce_physical_item_property_append_only` `function_search_path_mutable`, global `auth_leaked_password_protection`) are unchanged and unrelated. **No frontend nav link added** — the repo has no shared header/nav component and the PRD forbids building nav infrastructure in Phase J; `/admin/queue` is reachable only by direct URL for now.
**Why**: Three rationales. (a) **RLS carries the authorization burden, not app code.** The alternative — filter by `org_id IN (SELECT org_id FROM org_admins WHERE person_id = caller AND revoked_at IS NULL)` in the Next.js page — duplicates logic already encoded in `is_org_admin()` and creates two places to fix if the rules shift (e.g., when Phase K changes the admin-revocation flow). RLS is the single source of truth for "who sees which request"; the page just reads. This is the same pattern Phase E's `/api/signup` uses for org lookups (RLS on `org(s)` lets the service client read what the signup form needs without app-side filtering). (b) **One SELECT policy with three branches instead of two policies.** The PRD's suggested "add sysadmin fallback policy" is unnecessary because `is_system_admin()` is already unconditional — adding a second policy would just OR together with the first, and two policies are harder to reason about than one disjunction. The admin-less-org fallback is a natural consequence of sysadmins seeing everything, not a separate concept. (c) **In-code 403 on top of RLS, not instead of.** RLS alone would return `not_found` (zero rows) to unauthorized callers, which conflates "doesn't exist" and "you're not allowed to see it." The endpoint's explicit `is_system_admin()` + `is_org_admin()` RPC pre-check surfaces a clean 403, while RLS continues to enforce the underlying visibility — belt + suspenders, both required.
**Alternatives rejected**:
- **App-layer org filtering in `app/admin/queue/page.tsx`** (the alternative to RLS-backed authorization): duplicates logic across app + DB; any discrepancy between the page's JOIN to `org_admins` and the RLS policy's `is_org_admin()` call would create either data leaks or false negatives. Rejected outright — the PRD Phase J instruction explicitly calls for "scoped by RLS (not hand-coded filtering)."
- **Two separate RLS policies — one for org admins, one sysadmin-fallback for admin-less orgs**: the sysadmin branch already covers admin-less orgs via `is_system_admin()`, which returns true regardless of whether `org_admins` has any rows for the target org. A second policy would OR-combine identically. Rejected to keep the policy surface minimal.
- **Wrapping the approval's two UPDATEs in a SECURITY DEFINER RPC for atomicity**: would be cleaner but the PRD explicitly says "Using two separate supabase-js .update() calls is fine for v2 — note this in a code comment if used." We note it. Follow-up phase can fold into an RPC without user-visible behavior change.
- **Resend-backed email delivery in this PR**: Resend SMTP setup is a Phase O deliverable (PRD §6.2); wiring it half-done would leave partial infrastructure the next author has to clean up. Stub + TODO is the right scope boundary for Phase J. Decisions don't roll back on email failure either way, so the stub is functionally indistinguishable from a Resend call that silently fails.
- **Adding a shared header component with a `/admin/queue` link**: PRD explicitly says "if there's no nav system, skip — `/admin/queue` is still reachable by URL. Don't build nav infrastructure for Phase J." The repo has zero shared header component today; adding one to host a single link is out of scope.
- **Exposing other requesters' emails via an RLS-readable view over `auth.users`**: `auth.users` is Supabase-managed and a view over it for `authenticated` role would leak email addresses broadly. The queue page currently shows `"(email on decide)"` for other users' emails; the decide endpoint resolves the email server-side via service role for the notification. Phase O can add a `SECURITY DEFINER` RPC `admin_list_emails_for_persons(uuid[])` gated by `is_org_admin`/`is_system_admin` if surfacing emails in the queue becomes required.
- **DELETE policy for org admins** (letting org admins purge decided requests): would erase audit history. Sysadmin-only DELETE preserves the decision log while still allowing abuse-cleanup from the top.

## 2026-04-24 — Auth/tenancy Phase K: zero-member org bootstrap + resign-preserves-membership + RLS-backed self-resign

**Affects**: `"01_tenancy"."org_admin_requests"` RLS policies (`oar_select`, `oar_insert`, `oar_update`, `oar_delete`), `"01_tenancy"."org_admins"` RLS (new `org_admins_self_resign` policy), `app/api/request-org-admin/route.ts`, `app/api/decide-admin-request/route.ts`, `app/api/resign-admin/route.ts`, `app/admin/queue/page.tsx` (admin-requests section added), `app/admin/queue/AdminRequestRow.tsx`, `components/RequestAdminButton.tsx`, `components/ResignAdminButton.tsx`, migration `supabase/migrations/20260424000012_phase_k_org_admin_requests_rls.sql` on `Nex4.23.26` branch `auth-tenancy-v2`, AUTH-TENANCY-PRD v2 §9 Phase K + TEST-PLAN Phase K.
**Decision**: Shipped Phase K as four commits. (1) **RLS migration** idempotently drops+recreates the `oar_*` policies on `org_admin_requests` — SELECT = `person_id=current_person_id() OR is_system_admin()` (requester sees own rows; sysadmins see everything for the queue); INSERT = `person_id=current_person_id() AND status='pending'` (paired with the route-layer membership + zero-member-bootstrap gate); UPDATE USING = `sysadmin OR self` with WITH CHECK = `sysadmin OR (self AND status='withdrawn')` so requesters can only transition their own row to `withdrawn` while sysadmins decide approved/denied; DELETE = sysadmin only. Added a NEW policy on `org_admins` named `org_admins_self_resign` — UPDATE USING `person_id=current_person_id() AND revoked_at IS NULL` with WITH CHECK `person_id=current_person_id() AND revoked_at IS NOT NULL`. This lets an admin flip their own `revoked_at` from NULL → `now()` but CANNOT un-revoke (WITH CHECK requires NOT NULL) and cannot touch other admins' rows (USING narrows to self). All existing sysadmin-only policies on `org_admins` (insert/update/delete) remain — policies OR-combine. Added partial index `org_admin_requests_pending_requested_at (requested_at DESC) WHERE status='pending'` for queue ordering. (2) **`/api/request-org-admin`** (Node runtime, `createServerClient`+cookies): resolves caller Person via `user_id`, looks up the target org (404 if missing), checks membership via `Person's_org = orgId`. Membership exception: if caller is NOT a member, we run `SELECT count(*) FROM Person(s) WHERE Person's_org = orgId` and allow the request if the count is zero (PRD "zero-member org bootstrap" — approval will transfer the Person in). Check already-admin via `org_admins WHERE org_id+person_id AND revoked_at IS NULL` (409 `already_admin` if active). INSERT pending row; partial unique index `org_admin_requests_one_open` kicks duplicates, which the route surfaces as `{ok:true, alreadyPending:true}` with the existing request id — idempotent UX for repeat clicks. (3) **`/api/decide-admin-request`** (sysadmin-only, belt-and-suspenders): explicit `is_system_admin()` RPC check → 403 `not_sysadmin` for non-sysadmins (RLS on `org_admins` INSERT would silently fail, but 403 is cleaner); resolves caller Person id for `granted_by_system_admin_id` + `decided_by`; fetches request row; approval path uses `supabase.upsert({..., onConflict:'org_id,person_id'})` on `org_admins` with `granted_via='sysadmin_approval'`, `granted_at=now()`, `revoked_at=null` — this ON CONFLICT DO UPDATE un-revokes a previously revoked admin's row (re-grant). Zero-member bootstrap inside the approval: if `requester.Person's_org != request.org_id` AND `count(Person's in target org) = 0`, UPDATE Person's_org to request.org_id. Does NOT transfer if the target org already has members and requester isn't one (defensive — that branch shouldn't reach approval because the request endpoint would have 403'd at INSERT time, but if a sysadmin approves anyway we grant admin without a Person transfer). Then UPDATE request row with `status=decision, decided_at=now(), decided_by=callerPersonId, reason=<reason>` guarded by `.eq('status','pending')`. Denied path is just the request UPDATE. (4) **`/api/resign-admin`**: UPDATE `org_admins SET revoked_at=now() WHERE org_id=? AND person_id=callerPerson AND revoked_at IS NULL`. RLS `org_admins_self_resign` narrows to the caller's own row. Does NOT touch `Person(s).Person's_org` — caller remains a member. Returns `not_admin` 403 if no active row (including the case where RLS hid it). (5) **`/admin/queue` extension**: added second `<section>` "Admin requests" below the Member requests section. Batched Person + Org lookups now span both tables' `person_id`s / `org_id`s to avoid duplicate round-trips. New `AdminRequestRow` client component is a near-verbatim copy of `QueueRow` but posts to `/api/decide-admin-request` with its own error-message mapping for `not_sysadmin`. Both sections render the same empty-state pattern. (6) **`RequestAdminButton`** + **`ResignAdminButton`** components shipped for Phase M to import. RequestAdminButton props: `{orgId, orgSlug, isMember, hasPendingRequest}`. Hides button if `!isMember`, renders "Request pending" pill if `hasPendingRequest`, otherwise POSTs to `/api/request-org-admin`. ResignAdminButton props: `{orgId, orgName, onResigned?}`. `window.confirm()` before POST. Both are client components — Phase M will wire placement onto `/{orgSlug}` pages. (7) **DB-level tests executed** via `mcp__supabase-schema-config__execute_sql`: seeded `test_phase_k_OrgA` + `test_phase_k_EmptyOrg` (both with `active_div_levels=1` to satisfy `validate_person_hierarchy`) + Persons `test_phase_k Requester1` (in sentinel), `test_phase_k Member1` (in OrgA), `test_phase_k Admin1` (in OrgA). AUTH-T075 (member submits admin request → row created): INSERT `org_admin_requests(OrgA, Member1, pending)` → 1 row PASS. AUTH-T076 (duplicate blocked by `org_admin_requests_one_open`): second INSERT raises `23505` unique violation PASS. AUTH-T078 (zero-member bootstrap): verified EmptyOrg has 0 Persons + Requester1 in sentinel; inserted pending request; simulated approved decision (Person's_org transfer + org_admins upsert with `granted_via='sysadmin_approval'` + request status→approved); final SELECT shows Person's_org=EmptyOrg, granted_via=sysadmin_approval, revoked_at=NULL, request_status=approved PASS. AUTH-T079 (resign flow): inserted org_admins row for Admin1 in OrgA with `granted_via='manual_bootstrap'`; ran UPDATE `org_admins SET revoked_at=now()` filtered by person+org+revoked_at NULL; verified `active_rows_after=0` AND `Person's_org` unchanged (still OrgA) PASS. (8) **AUTH-T074 / T077 (sysadmin approve flows requiring a seeded sysadmin)**: SKIPPED per task instruction — DB currently has 0 seeded sysadmins and creating one would require an `auth.users` + Person + `system_administrators` seed that risks corrupting other phases' tests. Marked manual; the simulated approval in AUTH-T078 covers the SQL side effects. (9) **Cleanup**: DELETE'd all `test_phase_k_*` rows from `org_admin_requests`, `org_admins`, `04_entities.Entity(s)_members`, `04_entities.Entity(s)` (auto-created by `auto_create_solo_entity` trigger on Person INSERT — lives in `04_entities` not `01_tenancy`), `Person(s)`, `org(s)`; verified leak counts = 0 across all four tables. (10) **Security advisors post-migration**: no new findings; pre-existing INFO/WARN entries (`08_moments.moment_node(s)_*` RLS-no-policy, `public.touch_updated_at` + `public.enforce_physical_item_property_append_only` `function_search_path_mutable`, `auth_leaked_password_protection`) unchanged. **Typecheck**: `npx tsc --noEmit` clean post-commit. (11) **No Phase M work done**: per PRD scope, org pages don't exist yet; Phase K ships the endpoints + queue + two reusable button components so Phase M can wire placement without refactoring.
**Why**: Three rationales. (a) **Zero-member org bootstrap is a first-class request-admin branch, not an error state.** PRD §9 K.5 explicitly covers "approval also transfers Person to org" for pre-seeded orgs that have lost all members (or self-created orgs whose sole creator got deleted). The alternative — require sysadmin to manually INSERT a Person row first and then approve the admin request — is two administrative steps where one suffices. Membership is derived from `Person's_org` in this schema (no junction table), so the transfer is a single-column UPDATE. We gate the bootstrap strictly on `count(Person's in org) = 0` so the exception cannot be weaponized to hijack an active org — any existing member would block the branch. (b) **Resignation preserves membership.** A user who resigns as admin is making a governance decision, not a membership decision — they still work for the org; they just no longer want to administer it. Collapsing "resign admin" into "leave org" would surprise users (they'd silently lose access to their org's data) and tangle Phase K with Phase L's leave-org flow. Keeping the two operations decoupled also means sysadmins can re-grant admin without a separate re-join flow. (c) **Self-resign via an RLS policy, not a service-role endpoint.** The alternative — have `/api/resign-admin` open a service-role client and bypass RLS — would centralize the rule in app code and duplicate it if any future surface needs to let admins resign from a different route. By narrowing UPDATE access to `person_id = current_person_id() AND revoked_at IS NULL` with a WITH CHECK forcing the new `revoked_at IS NOT NULL`, the database itself enforces "only the admin themselves, only transitioning NULL → timestamp." The user-session client does the work; RLS is the authorization boundary. Same pattern as Phase J's admin queue: RLS-as-authorization. (d) **Upsert with ON CONFLICT DO UPDATE on `org_admins` for the re-grant path.** When a previously revoked admin re-requests and a sysadmin approves, we want a clean "un-revoke" that also stamps the new grant metadata (`granted_at`, `granted_by_system_admin_id`). `supabase-js`'s `.upsert({onConflict:'org_id,person_id'})` maps straight to `INSERT ... ON CONFLICT (org_id, person_id) DO UPDATE SET ...` which is both atomic and succinct. The UNIQUE constraint on `(org_id, person_id)` is already there from Phase A — we're just leveraging it. (e) **Route-level 403 for `not_sysadmin` on decide-admin-request plus RLS.** RLS alone on `org_admins` INSERT would cause a silent 0-row UPDATE for non-sysadmins; surfacing a clean 403 instead of a confusing 500 or success-with-no-effect is standard defence-in-depth and matches Phase J's decide-member-request pattern. (f) **Ship `RequestAdminButton` + `ResignAdminButton` even though Phase M doesn't exist yet.** Two tiny client components with defined props contracts are cheaper than forcing Phase M to re-discover the API shape. Shipping them means Phase M is a pure placement task.
**Alternatives rejected**:
- **Require sysadmin to manually transfer Person before approving a zero-member bootstrap request**: two admin clicks where one suffices; also fragile because there's no UI for "manually transfer Person" in v2 (Phase N adds unverified-org review but not generic Person ops). Rejected — the PRD explicitly specifies the combined atomic operation.
- **Service-role endpoint for `/api/resign-admin` that bypasses RLS**: concentrates authorization in app code; risks privilege escalation bugs (a future API that accepts a `person_id` parameter could let a sysadmin-less caller resign anyone). RLS-narrowed self-update is tighter and survives API-shape mistakes.
- **Un-revoke via a separate `/api/restore-admin` endpoint instead of upsert-on-decide**: adds a surface, a policy row, and a UI affordance that has no natural placement (sysadmin queue already IS the decision point for a re-request). Collapsing into the approved path is simpler.
- **Transfer Person AND grant admin even when target org has members but requester isn't one**: would let a sysadmin silently reassign any user's org membership through the admin-request queue — breaks the Phase L "leave org is a user decision" invariant. We grant admin without transferring Person in that edge case (defensive; normally the request endpoint would 403 at INSERT time).
- **Gate the zero-member bootstrap branch on `is_verified=false`**: would require pre-seeded orgs with zero members (a plausible import-cleanup state) to go through sysadmin intervention separately. The member-count check is a cleaner invariant — verification status is orthogonal.
- **Seed a synthetic sysadmin to run AUTH-T074/T077 in-test**: risks polluting other phases' tests that assume `count(system_administrators) = 0`. The simulated-approval SQL in AUTH-T078 exercises the same DB side effects; full HTTP E2E defers to manual testing once the genesis sysadmin is seeded.
- **Put the admin-requests section on a separate `/admin/admin-queue` page**: doubles the route surface and forces the sysadmin to check two URLs. Stacking as a second section on `/admin/queue` keeps the queue-of-queues concept intact.
- **Have `RequestAdminButton` fetch `hasPendingRequest` itself on mount**: Phase M's org page already knows this (it will query `org_admin_requests` server-side when rendering). Passing as a prop keeps the component a pure renderer.

## 2026-04-24 — Anti-enumeration on `/api/request-download-link`

**Affects**: `/api/request-download-link` (Phase I), returning-user magic-link flow, PRD §4.4 / §9 Phase I
**Decision**: The endpoint always responds with `{ok:true}` and HTTP 200 — regardless of whether the submitted email exists in `auth.users`, is malformed, or is missing entirely. A 100–300 ms random delay is injected on every request to blur the timing side-channel between the "user found → generateLink" path and the "no user → noop" path. `ReturningUserForm.tsx` mirrors this by showing the same "If we found your account, a sign-in link is on the way." copy regardless of response body.
**Why**: Distinguishable responses (different status codes, different messages, or a measurable response-time delta) would let an attacker enumerate which emails already have a Nexus account, which is both a privacy leak for existing users and a useful primer for targeted phishing.
**Alternatives rejected**: (a) Returning `404` when the email is unknown — reveals existence directly. (b) Returning `{ok:true}` but skipping the delay — keeps a timing side-channel that is trivially measurable with a handful of requests. (c) Rate-limit-only defence (Phase O, 3/hr per email) — useful but orthogonal; rate limiting slows enumeration, it does not prevent a single-shot lookup. (d) Server-side CAPTCHA — heavier UX cost than the 100–300 ms delay buys in signal reduction.

One-line change, no data to migrate (Phase B's trigger only just shipped and the FK's nullability survives the cascade semantics — pre-existing Persons with `user_id IS NULL` are unaffected). (b) Soft-resolve via `UPDATE pending_signups SET resolved_at=now()` instead of `DELETE` — the PRD draft says "delete pending_signups row (resolved)," but a soft-mark gives us an audit trail of who picked which org at signup time (useful when investigating "why is Jane in ExxonMobil instead of the gmail.com queue"), lets the domain-match branch verify retrospectively which domain rule matched on a given day, and lets the trigger's NOT-FOUND guard (filter `resolved_at IS NULL`) naturally block re-runs without a second ON CONFLICT path. Split_part on `lower(NEW.email)` is safe because `org_email_domains.domain` is CHECK-constrained to `domain=lower(domain)`.
**Alternatives rejected**:
- `DELETE FROM pending_signups WHERE auth_user_id=NEW.id` per PRD draft: loses the audit trail. A separate `pending_signups_resolutions` log would double the write volume and introduce a two-writes-atomic concern. Soft-resolve gets the same property for free. Deviation from the literal PRD wording is explicitly documented here per request.
- Keep `Person(s).user_id` FK at `SET NULL` and do manual `DELETE Person(s) WHERE user_id IN (...)` in test cleanup: six lines of imperative bookkeeping at every cleanup call-site versus a one-line FK flip. Also leaves an attractive nuisance for the next dev who writes a GDPR delete script.
- Create a `attempt_link_existing_person(org, auth_user)` no-op stub so Phase F can call it symmetrically: dead code. PRD §10 multi-org membership is out of scope; when/if it lands, Phase N (or whichever phase takes it on) will create the function and add the call site to this trigger. Skipping the call is cheaper than shipping a function with no body and no callers.
- Trigger `BEFORE UPDATE` so the function can short-circuit before the row is written: buys nothing — the body's idempotency guards already block duplicate work, and AFTER is the Postgres convention for side-effect-only triggers. Also keeps the NEW.email_confirmed_at reliably populated.
- Use `FOR EACH STATEMENT` instead of `FOR EACH ROW`: breaks the contract — one verified user = one route decision. Statement-level triggers don't get NEW.id per row.
- Fire unconditionally and do the NULL→NOT-NULL check inside the function: wastes a function call on every UPDATE to `auth.users` (profile edits, `last_sign_in_at` bumps, metadata patches). Postgres evaluates the WHEN clause cheaply at the row level without entering the executor.
- Look up `intent_org_id` from `auth.users.raw_user_meta_data` instead of `pending_signups`: PRD decided against that path in Phase A (stashing intent in metadata loses the `new_org_creator` flag indexability and leaves stale state after verification). Sticking with `pending_signups` as the source of truth.

- Skip the `SET search_path=''` hardening: Supabase's security advisor flags every SECURITY DEFINER function without a pinned search_path as a `function_search_path_mutable` WARN. Cheap fix, no semantic change.

- Track spawn provenance via a separate `recurring_task_spawns(recurring_task_id, moment_node_id, spawned_at)` log: duplicates information already on the moment_node row; reporting queries would need a JOIN they don't need today.

## 2026-04-23 — `"recurring_task(s)"`: scheduled task spawn tied to a Responsibility

**Affects**: `"06_templates"."recurring_task(s)"` (new), `public.enforce_recurring_task_invariants()` (new), triggers `recurring_task_s_invariants_trg` / `recurring_task_s_touch_updated_at`, reuse of Rule 15 `obligation_kind` ENUM and `public.touch_updated_at()`, migration `add_recurring_task`.
**Decision**: New `"06_templates"."recurring_task(s)"` table keyed `uuid`, with NOT-NULL FKs to `"responsibility(s)"` (cascade), `"node_template(s)"` (cascade), and `"Entity(s)"` (restrict) — all same-tenant, enforced by trigger. Cadence is modeled as a `cadence_cron text` (5/6-field cron expression) + `timezone text` default `'UTC'`, with a materialized `next_due_at timestamptz` and `last_spawned_at timestamptz` maintained by the spawner job (not a DB trigger — scheduling state changes frequently and the DB should not be the schedule authority). Optional `obligation obligation_kind` reuses Rule 15's shared ENUM (`recommended | mandatory | prohibitive`), optional `policy_id` FK to `"policy(s)"` for cases where the obligation traces to an authored rule. `is_active boolean` flag for pause-without-delete; `deleted_at` for soft-delete. Partial indexes on `(org_id)`, `(next_due_at)` are scoped to `WHERE deleted_at IS NULL AND is_active = true` so the spawner's "what's due?" query is a tight index scan. Standard audit block + `touch_updated_at` trigger. RLS: full CRUD per Rule 10 gated on `org_id = current_user_org()`. Spawner job (out of scope for this migration) reads `next_due_at ≤ now()` rows, inserts a `moment_node(s)` row of type Task with `spawned_from_recurring_task_id` set (Migration 7), updates `last_spawned_at` and advances `next_due_at` via the cron expression.
**Why**: Q4 locked "separate recurring_task table, not a cadence column on node_template" — cadence is not a structural property of the template (same template shape can be spawned one-off, scheduled daily, or scheduled weekly by three different recurring_task rows). Q2 locked "Responsibility ⊃ repeating task" — recurring_task.responsibility_id is NOT NULL because every recurring obligation IS a responsibility (you can't have an ongoing scheduled task with nobody accountable); the inverse — one-off tasks that don't trace to a responsibility (Q14) — is handled at the moment_node layer by a nullable `responsibility_id` there (Migration 7). `owner_entity_id` via `"Entity(s)"` keeps the Rule 8 universal-who pattern consistent with role assignments. Cron was chosen over Postgres `interval` / bespoke cadence columns because cron encodes natural scheduling patterns like "9am every weekday" in a single string that UIs can render visually and job engines (pg_cron, supervisor, k8s CronJob) consume directly. `next_due_at` materialized because the spawner's polling query "anything due?" should be an index scan, not a per-row cron-expression evaluation. Timezone column beside cron because cron expressions are timezone-sensitive ("9am" means different UTC moments in different zones); conflating them into cadence_cron would produce silent off-by-hours bugs twice a year at DST transitions. `ON DELETE RESTRICT` on `owner_entity_id` because deleting the owner while active recurring obligations exist is a governance issue; forcing the user to reassign first is the safer default (the cascade choice on responsibility/template is the opposite: killing the parent reasonably kills the children).
**Alternatives rejected**:
- Cadence columns on `"node_template(s)"`: Q4 rejected. One template shape may be spawned at different cadences by different recurring_task rows; putting the schedule on the template forces either one cadence per template (too restrictive) or nullable cadence columns on every template (wide table, many empty columns).
- `interval`-based cadence (`repeat_every interval`): expressive for "every 7 days" but cannot represent "9am every weekday except Sunday." Cron covers both cases.
- `obligation obligation_kind NOT NULL`: forces a classification even when none applies (ad-hoc recurring reminders aren't "mandatory" or "prohibitive"). NULL allowed keeps the column optional framing.
- No `policy_id` FK (obligation kind standalone): loses the link to the authored rule behind the obligation; when a policy changes, the recurring tasks it generated should be discoverable.
- Trigger-maintained `next_due_at`: requires pg_cron or similar inside the DB; puts the scheduling engine behind RLS. App-layer/job-layer maintenance keeps the DB as a passive store.
- Same-tenant enforcement via RLS only: RLS filters reads but does not prevent writes via service role; trigger is defense-in-depth.
- `ON DELETE CASCADE` on `owner_entity_id`: deleting an Entity would silently drop recurring obligations; surprising and bad for governance audits. RESTRICT makes the cleanup explicit.
- Include spawn history in this table (per-spawn rows): per-spawn record is the spawned `moment_node(s)` itself (via `spawned_from_recurring_task_id`). Duplicating that here would denormalize.

## 2026-04-23 — `public.effective_roles` view: union of direct assignments + title-bundled roles

**Affects**: `public.effective_roles` (new view), `public.person_solo_entity_id()` (consumer), `"04_entities".role_assignments`, `"06_templates".title_roles`, `"01_tenancy"."title(s)"`, `"01_tenancy"."Person(s)"`, migration `add_effective_roles_view`.
**Decision**: Created `public.effective_roles` as a `security_invoker = true` view that UNIONs direct role assignments with title-bundled roles resolved at read time. Direct arm: `SELECT entity_id, role_id, org_id, 'direct' AS source, assigned_at AS effective_since, id AS assignment_id, NULL::uuid AS title_id FROM "04_entities".role_assignments WHERE revoked_at IS NULL`. Bundled arm: for every active Person (`deleted_at IS NULL AND is_active = true`) with a non-null `title_id` (and a non-deleted title), join `title_roles` and emit `entity_id = public.person_solo_entity_id(person.id)`, `org_id = "Person's_org"`, `source = 'from_title'`, `effective_since = GREATEST(person.updated_at, title_roles.created_at)`, `assignment_id = NULL`, and `title_id = person.title_id` so the consumer can trace provenance. `GRANT SELECT TO authenticated` makes the view data-API accessible. `UNION ALL` (not `UNION`) because the physical-distinct pass is unnecessary at UI scale and DISTINCT-by-(entity_id, role_id) is the consumer's responsibility when they want it.
**Why**: Q18 locked derive-not-materialize for title-bundled role assignments — changing a title's bundle must instantly flip every holder's effective roles without a migration of `role_assignments` rows. A view unioning the two sources is the canonical way to deliver this semantic in Postgres: the JOIN to `title_roles` happens once per query, holders see new/removed bundled roles the instant the junction changes, and no triggers need to be audited for bundle-sync correctness. `security_invoker = true` (introduced in Postgres 15) is the correct mode for a view built on top of RLS-enabled tables — the view evaluates the underlying policies as the querying user rather than as the view owner, so a non-admin querying `effective_roles` sees exactly the rows their RLS context permits without the view having to re-implement those policies. Rule 8's solo-Entity pattern is what makes the bundled-arm cleanly yield `entity_id` — without it, the bundled arm would have to emit `(person_id, entity_id)` polymorphic rows, doubling the view's column count and forcing every consumer to branch on "is this a person-holding-role or an entity-holding-role." `effective_since` is the best-effort "since when did this hold" column for UIs that want to display a date; it's `GREATEST(p.updated_at, tr.created_at)` because either changing the person's title OR adding the role to the title's bundle counts as "started holding this effective role" — whichever is later wins. Not audit-grade because the true answer requires scanning `status_change_log` / history tables; a UI date is sufficient.
**Alternatives rejected**:
- Materialize bundled rows into `role_assignments` with a `source text` column and triggers keeping them in sync on title change / bundle edit: creates reconciliation complexity and "forgot to fire the trigger" class bugs. Rejected in Q18.
- Stored function `effective_roles_for(entity_id)` returning SETOF: fine for per-entity reads but forces a UDF call site into RLS policies and joins; a view is composable with filters and JOINs the normal way.
- Materialized view + REFRESH on writes: adds staleness window + refresh trigger complexity to avoid a perf problem that doesn't exist at the scale of role assignments.
- `security_definer` view: would bypass the querying user's RLS and force the view to replicate every upstream policy manually. Invoker mode is correct when the upstream RLS is already the right policy.
- `UNION` (distinct) instead of `UNION ALL`: Postgres' physical-distinct pass is pure overhead when consumers frequently don't need it (e.g. "does entity X hold role Y?" via EXISTS doesn't care about duplicates). Callers that need distinct can add it.
- Exclude from-title rows when a direct row exists for the same (entity_id, role_id): would hide the fact that a role is held via two paths; a reporting query may want to see both.
- Include revoked direct assignments with a `revoked_at` column in the view: adds one more column consumers must filter on, and revoked assignments are not "effective"; exclusion is the right default.
- Join `Person(s)` via `auth.uid()` rather than starting from `Person(s)`: would scope the view to the querying user only, blocking "show me everyone who holds Fire Warden" queries that the view must support.

## 2026-04-23 — Role assignments (`role_assignments`): Entity holds Role, same-tenant only, soft-revoke

**Affects**: `"04_entities".role_assignments` (new), `public.enforce_role_assignment_invariants()` (new), trigger `role_assignments_invariants_trg`, migration `add_role_assignments`.
**Decision**: New table `"04_entities".role_assignments` keyed `id uuid`, with `org_id uuid NOT NULL` (tenant tag), `role_id uuid` FK `"06_templates"."role(s)"` `ON DELETE CASCADE`, `entity_id uuid` FK `"04_entities"."Entity(s)"` `ON DELETE CASCADE` (Rule 8 universal target — solo-Entity for Person, named Entity for group). Rule 6 temporal soft-delete via nullable `revoked_at` + `revoke_reason`; partial UNIQUE `(role_id, entity_id) WHERE revoked_at IS NULL` enforces at-most-one-active-assignment per pair while allowing a revocation-then-reassignment history. Two extra partial indexes on `role_id` / `entity_id` (active only) cover the two dominant read patterns ("who holds this role?" / "what roles does this entity hold?"). A `BEFORE INSERT OR UPDATE OF (org_id, role_id, entity_id)` trigger `enforce_role_assignment_invariants()` (SECURITY DEFINER, wide search_path) asserts `role.org_id = entity.org_id = this.org_id` — same-tenant only. RLS: full CRUD per Rule 10, gated on `org_id = current_user_org()`. Placed in `"04_entities"` (runtime-side of template/instance split) rather than `"06_templates"` because an assignment is a who-holds-what fact, structurally closer to `"Entity(s)_members"` than to the authored role definition.
**Why**: Q12 locked "role assignments target Entity, not Person" — Rule 8's solo-Entity pattern means individuals and groups use the same FK column, collapsing what would otherwise be a polymorphic (person | entity) target into a single clean reference. Q18 locked derive-not-materialize for title-bundled roles, so this table stores ONLY direct/ad-hoc assignments; title bundles stay in the `title_roles` junction and are unioned at read time by `public.effective_roles` (Migration 5). Same-tenant enforcement matters despite Rule 8 allowing cross-tenant entity membership: a role is a per-org functional definition (Q17), and assigning Org A's "Fire Warden" to a member from Org B would create ambiguous accountability. Trigger-based enforcement is the right layer because a CHECK constraint cannot dereference foreign tables and app-layer enforcement is bypassable by direct SQL. `revoked_at` over `deleted_at` because assignments are temporal facts (Rule 6) — "who held this role during Q2?" is a valid audit question and hard-delete would lose that history. The partial active-unique index is the important single-active-per-pair constraint; it allows historical rows to accumulate without interfering with new assignments after revocation.
**Alternatives rejected**:
- `person_id` direct FK instead of `entity_id`: breaks the group-assignment case (Q12 choice B). Cross-linked with Rule 8: wrapping every Person in a solo-Entity was done precisely to avoid the polymorphic (person | entity) target shape; backing away from it here would reintroduce the very problem Rule 8 solved.
- Materialize title-bundled assignments as real rows with a `source` column (`direct | from_title`): creates reconciliation machinery (on-title-change triggers to diff-and-update) and a class of staleness bugs (edit the title's bundle, some holders never get the new row because the trigger fired before the bundle change). Q18 explicitly rejected this in favor of the view.
- Enforce same-tenant in RLS instead of via trigger: RLS can filter reads but cannot prevent writes that bypass policies (service role); the trigger fires on every write including service-role writes. Defense in depth.
- Hard delete on revoke (no `revoked_at` column): loses the "who held this during X incident?" audit trail. Historical assignment records are load-bearing for incident review.
- `started_at`/`ended_at` naming (temporal-interval style): conflicts with Rule 6's `revoked_at` convention used by `"Person(s)_contact_shares"` and `"Person(s)_contact_share_delegations"`. Consistency with existing pattern wins.
- Place in `"06_templates"` alongside `role(s)`: blurs the template/instance line that Rule 13 establishes — the authored role is the template; the assignment is the instance.
- `UNIQUE (entity_id, role_id)` (role as primary): functionally equivalent under cardinality but (role, entity) reads "who holds this role" which is the more common access direction.
- Cascade-restrict on `entity_id`: would block entity deletes while assignments exist; cascade + soft-revoke-on-delete pattern is the existing norm (e.g. `"Person(s)_contact_share_delegations"` cascades on entity delete).

- Redundant `org_id` columns on both junctions: violates single-source-of-truth (parent's org_id is authoritative); adds write tax without query benefit since RLS via EXISTS is equally fast with the index on `role_id` / `title_id`.
- Separate `position` sequences per junction (globally ordered across an org): unnecessary — position is within-parent (rank of responsibilities within a role, or roles within a title).

**Decision**: Linked the existing template/instance machinery (Rule 13) to the phase-2–6 catalog and physical-item worlds by three additions all in `"06_templates"`. (1) New nullable FK column `"process_template(s)".product_service_id → "10_catalog"."product_service(s)"` with `ON DELETE SET NULL`. No CHECK on `kind`: products AND services both get recipes (user brief — products because "many physical items go into building those products"; services as originally specified). Multiple templates per `(org_id, product_service_id)` allowed for variants (Standard / Premium / Small-Site / Large-Site). (2) `process_template_problems` N:N junction declaring which problems a recipe targets — `(process_template_id, problem_id)` UNIQUE, optional `position smallint` for org-internal preference ranking, optional `notes text`. Enables the `service × problem → recipe` routing query at discovery time: customer picks a service, states their problem, platform finds recipes targeting that problem. (3) `"node_template_physical_requirement(s)"` step-level physical-input declaration with three nullable FKs and a CHECK enforcing exactly-one-populated: `required_product_service_id` (a specific catalog product type), `required_category_id` (any physical item of this category), or `required_physical_item_id` (this specific asset). `quantity smallint NOT NULL DEFAULT 1`, optional `notes`. A BEFORE INSERT/UPDATE trigger enforces (a) when `required_product_service_id` is set, the target's `kind` must be `'product'` — you don't "use" a service as a physical input; and (b) when `required_physical_item_id` is set, the item must share `org_id` with the template. Runtime binding to `"moment_node(s)"` physical usage is left to the instance layer (phase-9 territory if it's ever needed). All three additions gate writes via `EXISTS` checks against the parent template's `org_id`.
**Why**: A catalog row + a recipe + a problem is the minimum structure to deliver the discovery UX the user described: customers pick a simplified service (catalog), state their problem (taxonomy + facet match), the platform surfaces recipes from orgs that offer that service and target that problem (junction), and recipe steps declare what physical assets are involved (requirement junction). Letting recipes declare problems explicitly instead of inferring from case studies scales O(N) with recipe count (vs O(N²) edge declarations between problems) and lets the UX rank recipes without requiring at least one completed case. The triple-nullable physical requirement models three realistic UX flows — "must be a Model 3000 Pump" (catalog-specific), "any truck will do" (category-generic), "must use truck #42" (bespoke pin) — without forcing the author to choose a shape. Keeping physical refs at the `node_template(s)` level (not `moment_node(s)`) lets templates remain reusable per-tenant without binding to a specific asset at design time; Rule 13's copy-on-spawn materialises the actual asset at instance creation.
**Alternatives rejected**:
- Recipe → single-product-or-service link: lost flexibility; some recipes legitimately span multiple products (a maintenance recipe for a pump that also involves consumables catalog rows). But this would be `"node_template_physical_requirement(s)"`-level, not `process_template`-level; separation of concerns holds.
- Services-only recipes (no recipes on products): user explicitly rejected — "products have recipes. Many physical items go into building those products." Manufacturing recipes are a real use-case.
- Recipe ↔ problem many-to-one (one problem per recipe): a single recipe may legitimately solve multiple related problems (scale buildup + rust removal from the same high-pressure wash); forcing one problem fragments recipes.
- Physical node as a 5th `"moment_node(s)"` partition: rejected in Q7 — physical items persist across time, moment nodes are time-stamped events; mixing them requires contorting the partition semantics and cascades constraint changes across every FK-into-moment-node site (Rule 11).
- Physical requirement as JSONB on `"node_template(s)"`: no FK integrity, no per-requirement attestability, no queryability for "which templates require a truck?" across tenants.

## 2026-04-23 — Physical item histories: attachment (temporal), property (append-only), assignment (temporal)

**Affects**: `"physical_item(s)".physical_item_level1_no_parent` CHECK (relaxed), `"physical_item_attachment(s)"`, `"physical_item_property(s)"`, `"physical_item_assignment(s)"` (all new in `"11_physical"`), Rule 14 (reused), migration `create_physical_item_histories`.
**Decision**: Three separate history patterns for three semantically different change types, not one unified log. (1) `"physical_item_attachment(s)"` records temporal parent-child relationships: `(parent_id, child_id, org_id, attached_at, detached_at, detach_reason)` with CHECK `parent_id <> child_id` and `detached_at >= attached_at`. Partial UNIQUE `(child_id) WHERE detached_at IS NULL` enforces at-most-one-active-parent per part. A BEFORE INSERT/UPDATE trigger validates same-org parent+child, matching `org_id`, and `child.level = parent.level + 1`. The `"physical_item(s)".parent_id` self-FK caches the current parent as a read-optimization; attachment rows are the append-only source of truth. Also relaxed `"physical_item(s)".physical_item_level_parent_chk` to `physical_item_level1_no_parent` (level=1 still has no parent; level>1 may be currently detached). (2) `"physical_item_property(s)"` adopts Rule 14 append-only supersession for attribute changes (paint color, condition, serial): `(item_id, org_id, property_name, property_value, value_captured_at, superseded_by_id, superseded_at)` with a BEFORE UPDATE trigger rejecting changes to anything except the two supersession columns. Partial index `(item_id, property_name) WHERE superseded_by_id IS NULL` serves the "current value" query. DELETE is sysadmin-only to preserve history. (3) `"physical_item_assignment(s)"` records Entity ownership over time: `(item_id, entity_id, org_id, assigned_at, unassigned_at, assignment_reason)`, partial UNIQUE `(item_id) WHERE unassigned_at IS NULL`, trigger enforcing `entity.org_id = item.org_id = this.org_id` (cross-tenant assignment blocked at DB layer). All three are full-CRUD RLS gated on `org_id = current_user_org()` per Rule 10, except `"physical_item_property(s)"`.DELETE which is sysadmin-only.
**Why**: The three change types differ structurally and semantically. **Part attachment** is a relational fact (a tire goes on a truck) — the natural shape is a two-sided temporal edge. A unified "change_log" can't easily express the at-most-one-current-attachment invariant via a partial unique index. **Attribute history** (paint, condition) is a per-row value change — Rule 14 already exists for exactly this on `"evidence_property(s)"`, so reusing it keeps the pattern count down. **Entity ownership** is also temporal (Fleet 1 → Fleet 2) but semantically distinct: it's not a physical configuration change, it's an ownership/accountability change. A separate table lets audit queries narrowly ask "who owned this pump when it broke?" without searching through unrelated attachment and property rows. Cross-tenant assignment block at the trigger layer is load-bearing: `"Entity(s)_members"` allows cross-tenant membership (Rule 8), but transferring a physical asset across tenants is out-of-scope for this schema and rejecting it at INSERT avoids silent tenant-boundary violations via Entity membership tricks.
**Alternatives rejected**:
- One unified `physical_item_change_log` with a `change_kind` discriminator: breaks the partial-unique-index invariants (can't express "tire is on at most one truck at a time" in an append-only log). "Current state" queries become replay-history instead of filter-active-rows. Rejected in Q9.
- Rule 7 "N active + archive via BEFORE UPDATE" for attributes: fits when there's a slot count (emails: max 2); attributes don't have slot semantics, and a new row per new value is more natural than mutating a status column.
- Temporal attachment + JSONB property history on the main row: JSONB loses FK integrity, append-only enforcement, and indexed current-value lookups. Rule 14 already solved this problem; reuse.
- Entity-assignment cascading on entity delete: would lose historical ownership record ("who owned it in Q2 last year?" loses meaning if the entity row is gone). `ON DELETE RESTRICT` on `entity_id` preserves history at the cost of requiring entity-delete to null out active assignments first.
- Keeping the strict `level>1 requires parent` CHECK and modeling detached parts via soft-delete: a detached tire isn't deleted, it's just currently unmounted; soft-delete conflates "gone" with "not currently attached."

## 2026-04-23 — Physical items: single-table tree with self-FK, level 1..3, global category taxonomy

**Affects**: `"11_physical"` schema (new), `physical_item_categories`, `physical_item_l1_subcategories`, `"physical_item(s)"`, Rule 16 schema map, Rule 10 (applied to physical_items), migration `create_physical_items_core`.
**Decision**: Created an eleventh numbered schema `"11_physical"` for tenant-owned physical assets. The tree is a single table `"physical_item(s)"` with a self-FK `parent_id`, `level smallint CHECK BETWEEN 1 AND 3`, and a composite CHECK `(level = 1 AND parent_id IS NULL) OR (level > 1 AND parent_id IS NOT NULL)` (later relaxed in phase 6 to support detached parts). Categories use a single `physical_item_categories` table with a `level` discriminator, Rule 17 moderation, partial UNIQUE `(level, lower(name)) WHERE status = 'published'`, and per-level starter seeds. L1 categories also support a subcategory tier via `physical_item_l1_subcategories` (trigger `enforce_l1_subcat_parent_level` enforces `parent_category.level = 1`; also Rule 17 moderated). `"physical_item(s)"` carries direct FKs for `category_id` (must match the item's level, trigger-enforced) and `l1_subcategory_id` (only populated when level=1, CHECK-enforced); no dual-reference tag junction needed because physical items are tenant-private so cross-tenant display of proposed categories is not a concern. An optional FK `product_service_id → "10_catalog"."product_service(s)"` links the asset to a catalog product (trigger enforces `kind = 'product'`, not service). Full CRUD RLS per Rule 10 gated on `org_id = current_user_org()` for the item table; categories use Rule 17's three-way status-branched RLS. A BEFORE INSERT/UPDATE invariants trigger enforces parent.level=this.level-1, parent.org=this.org, category.level match, and catalog kind.
**Why**: User chose a single self-FK table over three separate L1/L2/L3 tables in Q8 — three levels of "the same kind of thing" (a physical object with name / serial / owner / condition) don't need structural separation, and self-joins for "all parts of this truck" are one query instead of a UNION-across-three-tables. The `Div1..Div5` precedent argued for separate tables, but those have heterogeneous semantics per level (Div1 is root-under-org, Div5 is leaf); physical items are homogeneous across levels, only the category taxonomy differs. Global categories (user said yes in Q17) enable cross-tenant queries — "how many Model 3000 pumps are in the platform?" — without requiring per-tenant category reconciliation. L1 subcategory tier matches user's 15×7 brief; L2 and L3 flat-15 matches user's L2 examples (interior/exterior/frame/chassis — no subcats mentioned). Rule 17 on categories is the anyone-proposes-sysadmin-publishes model from Q2, adapted per-level. Catalog FK with kind=product trigger gives "this pump is a Model 3000" semantics (user's Q15 choice) without forcing every physical item into the catalog — generic items (a random truck) just carry `product_service_id = NULL`.
**Alternatives rejected**:
- Three separate tables `physical_item_l1`, `physical_item_l2`, `physical_item_l3`: preserves type-safety per level at the cost of triple-joining cross-level queries. User rejected in Q8 — physical items are genuinely uniform in shape across levels.
- Recursive `parent_id` self-FK with no level column (arbitrary depth): user chose hard 3-level cap in Q8; arbitrary depth invites category-schema drift as "where does a sub-sub-component go?" becomes a per-tenant judgment call.
- Per-tenant categories: user wanted both per-tenant AND global, but explicitly landed on global for canonical + per-tenant for org-specific overlays. Tenant overlay lives in `"12_overlay"` (phase 8) as tag junctions rather than a second category table.
- Dual-reference tag junction for category assignment (Rule 17 standard): overkill for tenant-private items where proposed categories never cross tenant boundaries. Direct FK + trigger is simpler.
- Required `product_service_id` (every physical item must map to a catalog row): too strict — one-off assets, prototypes, and loaner equipment wouldn't have catalog entries.
- `product_service_id` with no kind CHECK: would allow "physical item that's an instance of a service" — a category error.

## 2026-04-23 — Problem cases + normalized 3-dim ratings (effectiveness / cost / safety)

**Affects**: `"problem_case(s)"`, `problem_case_solutions`, `problem_case_ratings`, `public.rating_dimension` ENUM (new), `enforce_rating_case_solution_match()`, migration `create_problem_cases_and_ratings`.
**Decision**: Added three tables in `"10_catalog"` for the case-study layer. (1) `"problem_case(s)"` is open-author per Rule 17 lifecycle (`proposed → published | rejected`) but with a relaxed write policy: while `status = 'proposed'`, the proposer's org can UPDATE/DELETE their own cases; once published, only sysadmin can touch. Columns: `problem_id` (FK RESTRICT), `reporter_entity_id` (FK `"Entity(s)"` — solo-Entity pattern), `narrative text`, `occurred_at timestamptz`, full Rule 17 lifecycle block, standard audit with `updated_at` maintained by `public.touch_updated_at()`. (3) `problem_case_ratings` is a normalized ratings table with `dimension` drawn from a new `public.rating_dimension` ENUM (`effectiveness | cost | safety`), `score smallint CHECK BETWEEN 1 AND 5`, optional `comment text`, `rated_at timestamptz`. Two partial unique indexes enforce "one rating per (case, dimension)" and "one rating per (case, solution, dimension)" separately. A BEFORE INSERT/UPDATE trigger validates `case_solution_id.case_id = this.case_id` so ratings can't point at foreign cases' solutions. Writes on solutions and ratings delegate to the parent case's proposer-org-while-proposed policy via EXISTS.
**Why**: Normalizing the 3 dimensions into a table (vs 6 columns on `problem_case(s)`) protects against a future 4th dimension ("time to resolve") — it's a row insert, not a schema migration that touches every existing case. The `public.rating_dimension` ENUM is the Rule-15-style shared vocabulary; it could later be reused by product-level ratings or service-level ratings without recreating the value set. Nullable `case_solution_id` is the lightweight form of Rule 12 polymorphism when the target set is just two (case OR solution-within-case); full `target_kind` fan-out is overkill. Q4 locked "C — both case-level required + per-product optional"; the partial-unique-index scheme implements that exactly. Open-author cases with proposer-org-while-proposed writes is the correct privacy story: the reporter's org can draft and iterate freely until ready, then sysadmin freezes the record. Unlike canonical problems, case-study moderation mostly accepts — rejection is reserved for spam or policy violations, not content curation.
**Alternatives rejected**:
- 6 flat columns on `problem_case(s)` (effectiveness_score, effectiveness_comment, cost_score, cost_comment, safety_score, safety_comment): works until a 4th dim is added, then you're migrating every case row and rewriting every query. Rejected in Q4.
- Full Rule 12 polymorphic `target_kind + target_id` on ratings: only two targets (case, case_solution), not worth the extra column and trigger. Nullable `case_solution_id` covers it.
- Case-level only (Option A in Q4): loses the "which part of the solution actually carried it" signal for multi-product bundles.
- Per-product-within-case only (Option B): loses the headline number for aggregations; forces averaging across products which is noisy when one product was a minor add-on.
- Case-level write policy open to anyone in the same tenant (not just proposer's org): expands the edit footprint beyond what was asked for; proposer's own org is the conservative default.
- Rejecting published cases (immutable after publish, no UPDATE even by sysadmin): too strict — sysadmins need to fix typos and redact PII surfaced later.

## 2026-04-23 — Problems: canonical table + facet-dimension similarity (severity / cause / domain)

**Affects**: `"problem(s)"`, `problem_categories`, `"problem_category_tag(s)"`, `problem_dimensions`, `problem_dimension_values`, `"problem_dimension_tag(s)"`, `enforce_rule17_dual_ref_problem_category()`, `enforce_rule17_dual_ref_problem_dim()`, migration `create_problems_canonical_and_dimensions`.
**Decision**: Problems are a canonical sysadmin-curated table in `"10_catalog"."problem(s)"` with full Rule 17 lifecycle. Relatedness between problems is modeled via **multi-dimensional facets** rather than explicit problem-to-problem edges or an embedding index: three seeded dimensions (`severity`, `cause`, `domain`) each with their own value vocabulary, and a `"problem_dimension_tag(s)"` junction letting a problem carry multiple values per dimension. Each facet tag uses Rule 17's dual-reference pattern (`value_id` + `fallback_value_id`), with a trigger additionally enforcing that both values belong to the tag's `dimension_id`. Secondary: a primary-category axis (`problem_categories` + `"problem_category_tag(s)"`) exists for the "categorization" the user specified, also Rule 17 + dual-reference. Seeded values: severity = (minor | moderate | serious | critical); cause = (mechanical | chemical | thermal | electrical | environmental); domain = (facility | equipment | process | personnel). Discovery similarity is a COUNT of shared dimension-value matches between two problems.
**Why**: The user's paper-cut-vs-water-cut-vs-broken-bone example demonstrates that relatedness is multi-axial: paper and water cuts share *mechanism* (cut), water cuts and broken bones share *severity* (serious injury). A category tree handles one primary axis but can't express "similar on X but not on Y"; explicit pair-wise `problem_relations` edges scale O(N²) with manual annotation and don't compose into ranked "most similar across these dimensions" queries. Faceted classification is the right mechanic: O(N) annotation per problem (one tag per dimension it's positioned on), and overlap-counting at query time gives ranked similarity that the UI can filter on any single dimension. The dual-reference tag + per-tag dimension validation is the same Rule 17 machinery as category tags, with an extra integrity check: `value_id.dimension_id = tag.dimension_id` (otherwise a user could tag a problem with a severity value under the cause dimension). Three starter dimensions with ~4 starter values each is the minimum viable set; Rule 17 lets it grow via user proposals without a cap. Problems themselves are sysadmin-curated (Q3 locked) because taxonomy drift ("HydroBlast" vs "hydro-blasting") matters more at the problem level than at the case level — case studies are many; canonical problems are the few stable handles they aggregate under.
**Alternatives rejected**:
- Explicit `problem_relations` table (problem_a, problem_b, relation_kind, strength): O(N²) manual curation; doesn't scale past ~50 problems. Rejected in Q20.
- Single `category` per problem: user's example (cuts similar on mechanism, cuts+breaks similar on severity) is inexpressible with one primary axis.
- Embedding/vector similarity via pgvector: automates problem-to-problem distance from names/descriptions but loses dimension-specific filtering ("similar in severity"). Could be additive later but isn't a substitute for faceted classification here.
- Dimensions fixed at 3 forever, no proposed new dimensions: user wanted growth — "cause, domain, all appropriate categories" implied "more dimensions come later." Rule 17 lifecycle on `problem_dimensions` allows it.
- No fallback on `"problem_dimension_tag(s)"`: cross-tenant display of a proposed value becomes invisible, breaking the cross-tenant aggregation query that is the whole point of problem discovery. Dual-reference keeps aggregates complete.

## 2026-04-23 — Catalog layer: `"product_service(s)"` in new `"10_catalog"` schema, kind discriminator (not partitioned), Rule 17 throughout

**Affects**: `"10_catalog"` schema (new), `"product_service(s)"`, `product_service_categories`, `"product_service_category_tag(s)"`, `public.touch_updated_at()` (shared helper), `public.enforce_rule17_dual_ref_ps_category()`, Rule 16 schema map, migration `create_catalog_schema_and_product_service`.
**Decision**: Created the tenth numbered schema `"10_catalog"` for the global cross-tenant catalog layer. The main table `"product_service(s)"` holds products and services in a **single table with a `kind text CHECK IN ('product','service')` discriminator** (not Postgres LIST-partitioned). Each row carries the full Rule 17 lifecycle (`status`, `proposed_by_org_id`, `proposed_by_person_id`, `published_at`, `rejected_at`, `rejection_reason` + status/date CHECK), standard audit block with `updated_at` maintained by a new shared `public.touch_updated_at()` trigger function, and a partial UNIQUE `(kind, lower(name)) WHERE status = 'published'` (dup names tolerated only during moderation). `product_service_categories` is a single table with a `kind` column separating product categories from service categories, Rule 17 lifecycle, partial UNIQUE `(kind, lower(name)) WHERE status = 'published'`. Seeded with 5 generic starters per kind (Equipment/Consumables/Parts/Tools/Chemicals + Installation/Maintenance/Inspection/Cleaning/Engineering), all `status='published'`, `proposed_by_org_id=NULL`. `"product_service_category_tag(s)"` is the many-to-many junction with Rule 17 dual-reference (`category_id` + `fallback_category_id`, enforced by trigger `enforce_rule17_dual_ref_ps_category`). Tag SELECT is open to all authenticated (tags on published rows are public); tag writes gated to the tagger's own org.
**Why**: A single table with a discriminator is the right tradeoff here — the two kinds share ~90% of their columns (name, description, categories, recipe linkage, offerings), Postgres LIST partitioning is overkill at the <10k-rows-per-tenant scale expected for a catalog (compared to `"moment_node(s)"`'s tens-of-millions), and splitting into two tables would force UNION queries on every cross-kind discovery. The catalog rows themselves need Rule 17 moderation (user confirmed Option B in Q2 — anyone submits, sysadmin publishes) because the catalog is cross-tenant visible and a free-for-all write would produce duplicate entries and vendor-spoofed listings. Dedicated `"10_catalog"` schema keeps the global discovery layer findable and isolates its RLS semantics from the tenant-owned schemas. Seeding with 5 conservative generic categories per kind provides a starting point without putting opinionated domain-specific names (like "Hydroblasting Nozzles") into every tenant's DB; sysadmin can expand via the same Rule 17 flow end-users will use. `public.touch_updated_at()` is a shared helper because every new `updated_at`-bearing table will need it; defining it once in public keeps function count down.
**Alternatives rejected**:
- Two separate tables `products` and `services`: UNION cost on every cross-kind query, twice the RLS policy set, schema drift risk as the "shared" columns diverge.
- Postgres LIST partitioning on `kind`: appropriate at tens-of-millions of rows (`"moment_node(s)"`); overkill for <10k rows, adds partition-key-in-PK tax without query-plan benefit.
- Place catalog in existing `"03_metadata"`: misleading — catalog is user-facing product data, not internal metadata about the DB.
- No per-kind UNIQUE (allow duplicate "Hydro Blasting" service names): first-mover-wins taxonomy results in disputes and spoofing; published uniqueness is the standard rule.
- Hard 15-row cap on categories (pre-Rule-17 pattern): user explicitly rejected in the Q21 discussion that produced Rule 17.
- Per-tenant category tables: duplicates the cross-tenant discovery vocabulary, which is the whole reason the catalog exists.

## 2026-04-23 — Rule 17: moderated seeded catalogs replace hard row caps (starting with `org_types`)

**Affects**: `org_types`, House Rule 17 (new), `enforce_org_types_max_rows()` (dropped), `org_types_max_rows_trg` (dropped), migration `rule_17_foundation_retrofit_org_types`. Supersedes the "Capped at 10 rows via BEFORE INSERT trigger" clause of the 2026-04-22 "Org classification via sysadmin-managed global `org_types` catalog; exactly-one per org" entry — the rest of that entry (global scope, `org_type_id` NOT NULL, ON DELETE RESTRICT, seed rows) stands unchanged.
**Decision**: Introduced House Rule 17 — global lookup tables use a `proposed → published | rejected` moderation lifecycle in place of a hard row cap. Retrofitted `org_types` as the first adopter in a single migration: dropped `org_types_max_rows_trg` and `enforce_org_types_max_rows()`; added six lifecycle columns (`status` with CHECK `IN ('proposed','published','rejected')`, `proposed_by_org_id`, `proposed_by_person_id`, `published_at`, `rejected_at`, `rejection_reason`) with FKs and a status/date consistency CHECK; backfilled the three seed rows to `status='published', published_at=created_at`; flipped the column default so future user-authored rows start as `'proposed'`; replaced the four sysadmin-only RLS policies with a three-way status-branched set (SELECT: published visible to all authenticated + proposer-org visibility on proposed/rejected + sysadmin-all; INSERT: any authenticated user may submit a `'proposed'` row where `proposed_by_org_id = current_user_org()` + sysadmin escape hatch; UPDATE / DELETE: sysadmin-only). Rule 17 also canonicalises the **dual-reference tag** pattern for junctions that tag user data with potentially-proposed lookup values — tag row carries both `{value}_id` (intent) and `fallback_{value}_id` (required when the primary is still proposed, must reference a published row); display resolves to `{value}_id` when its status is `'published'`, otherwise to `fallback_{value}_id`. The pattern is reused across every category / dimension / subcategory table in the imminent product–service–problem–physical-item build-out (phases 2–8).
**Why**: The user brief requires (1) a global catalog of products / services / problems visible cross-tenant, (2) anyone-can-propose authoring with sysadmin moderation, and (3) the proposer can use their own pending value immediately — displayed publicly via a published fallback until approved, automatically upgraded on approval, cleanly falling back on rejection. A hard 15- or 7-row cap on categories collides with "anyone can submit for addition" because a cap blocks submissions once saturated and its ceiling is either under-provisioning (legitimate new values exceed it) or decorative (large enough that it's never hit). The `proposed → published` lifecycle preserves curation quality (sysadmin decides what is canonical) without constraining organic growth. Capturing this as a House Rule up front ensures consistent implementation across the ~10 tables that adopt it — otherwise each phase would re-derive the RLS shape and drift. `org_types` was chosen as the proof-of-pattern because it already existed with the old cap pattern, so the retrofit is both a migration of a real table and the first concrete worked example of the new rule.
**Alternatives rejected**:
- Keep the 10-row `org_types_max_rows_trg` cap: incompatible with "anyone can propose" — a proposer would hit the ceiling and be blocked from contributing even legitimate entries. The cap was authored for a world without user-submitted proposals.
- Per-table bespoke moderation RLS (no shared House Rule): fine for one table but produces drift across 10+ implementations. A stated rule is grep-able and forces consistency on the dual-reference tag shape, the status/date CHECK, and the SELECT/INSERT policy pair.
- `is_published boolean` instead of a three-value enum: drops the `'rejected'` state so a proposer cannot see *why* their submission didn't land, and `rejection_reason` needs a corresponding status to hang off. Enum gives proposers feedback without reading sysadmin channels.
- Pure community moderation with no sysadmin gate (wiki-style): produces taxonomy pollution (duplicate "HydroBlast" / "hydro-blasting" / "Hydro Blast" entries) without a reconciliation authority. The sysadmin transition is the reconciliation point.
- Deferring Rule 17 to the first category-table phase (phase 2+): `org_types` would continue using the hard cap indefinitely or be retrofitted later. Landing the pattern on `org_types` first pays the migration cost once against a 3-row table and produces a reference implementation before the larger category tables land.
- Tag table without `fallback_{value}_id` (single-column reference to lookup): if the referenced value is still `'proposed'` and only visible to the proposer, cross-tenant display has no value to show and users filtering / aggregating over tags would silently exclude the proposer's data. Requiring a published fallback while the primary is proposed keeps cross-tenant aggregates complete.

**Why**: User needed LinkedIn URL, past employers with titles, and per-person freeform notes, plus the ability to share each of these separately with specific Entities (own Org, any Div1..5, internal or external Entities — all reachable through the Rule 8 Entity-wrapping pattern). Notes use a different privacy model because their default ("only the writer") differs from contact rows' default ("whole org") — a tier on notes would be misleading.

- Universal "property moment node" subtype housing LinkedIn, employment, and notes under one polymorphic schema: loses typed columns, makes simple queries ("read this Person's LinkedIn") verbose JSON lookups, and drags in Rule 11 partitioning for fields that don't need it. User explicitly chose to avoid these costs once named.
- FK `employment_history.employer_name → "org(s)".name`: forces loading non-tenant historical employers into the tenant-owned `"org(s)"` table.
- Note visibility encoded as a 4th tier value (e.g. `'author_only'`): muddies Rule 5 for no gain, since notes would never use the other three tier values.
- RLS auto-granting the subject SELECT on notes written about them: rejected per user choice 2026-04-23 — matches standard HR/CRM behavior where the subject does not see notes unless explicitly shared.

## 2026-04-22 — Org classification via sysadmin-managed global `org_types` catalog; exactly-one per org

**Affects**: org_types, org(s).org_type_id, enforce_org_types_max_rows(), migrations `add_system_admins_and_org_types` + `org_type_id_not_null`
**Decision**: `org_types` is a **global** (not per-tenant) lookup table keyed on `id bigint identity` with `name text NOT NULL UNIQUE`. Seeded with three rows: `Contractor`, `Facility`, `Product/service provider`. Capped at 10 rows via BEFORE INSERT trigger `org_types_max_rows_trg` → `enforce_org_types_max_rows()`. `"org(s)"` gains `org_type_id bigint NOT NULL` with FK to `org_types.id` `ON DELETE RESTRICT`. The two existing orgs (Acme Corp, TestCo) were backfilled to `Facility` before the NOT NULL was promoted (split across two migrations: add the nullable column + FK, backfill, then `org_type_id_not_null`). RLS on `org_types`: SELECT open to any `authenticated` user; INSERT / UPDATE / DELETE gated by `is_system_admin()`.
**Why**: User framed the feature as "default org type configurable by the System Administrator only" and "every org must be one of these types" — that's a single source of truth managed outside the tenancy boundary, and a required single classification per org. Exactly-one cardinality (single FK column, not a join table) matches "must be one of" and keeps org-level queries trivial. Global scope (no `org_id` on `org_types`) matches "configurable by the System Administrator" — one admin, one list, not per-tenant customization. `ON DELETE RESTRICT` is the only FK action consistent with `NOT NULL`: `CASCADE` would delete orgs (catastrophic), `SET NULL` would violate the column constraint. The 10-row cap is enforced by a trigger because Postgres `CHECK` constraints are row-local and can't see `count(*)`; a trigger is the standard workaround and runs cheaply BEFORE INSERT.
**Alternatives rejected**:
- Per-tenant `org_types` (with `org_id` column): contradicts "sysadmin configures them" — there's one sysadmin and one list by user's framing.
- Many-to-many tagging (join table, no direct column on `"org(s)"`): more flexible but rejected — user wants a primary classification, not a multi-tag. A junction can be added later without disturbing the required-primary column.
- Leave `org_type_id` nullable indefinitely: user wanted every org to *have* a type; NOT NULL is the enforcement point.
- Enforce the 10-row cap in app code only: bypassable by direct SQL and by any future admin tool; trigger makes the invariant authoritative.
- `ON DELETE CASCADE` on the FK: would delete orgs when a sysadmin deletes a type — unacceptable blast radius.

## 2026-04-22 — System admin role: dedicated `system_admins` table keyed on `user_id`

**Affects**: system_admins, is_system_admin(), org_types RLS, House Rule 9, migration `add_system_admins_and_org_types`
**Decision**: System administration is modeled as a dedicated table `system_admins` with `user_id uuid PRIMARY KEY` FK to `auth.users.id` `ON DELETE CASCADE`. The membership check is wrapped in a SECURITY DEFINER helper `is_system_admin()` (House Rule 9) so other tables' RLS can call it without recursing into `system_admins`'s own RLS. `system_admins` has a single RLS policy — SELECT gated by `user_id = auth.uid()` (a user can see their own row). No write policies for the `authenticated` role; the table is bootstrapped and maintained via direct SQL under the service role, making sysadmin elevation a deliberate out-of-band act.
**Why**: "System Administrator" is cross-tenant by definition — a role that exists above the tenancy boundary. Representing it as a flag on `"Person(s)"` would couple the role to tenant membership, which inverts the semantic (sysadmins don't belong to an org; they manage across orgs). A dedicated table keeps the concept grep-able and visible in the schema. `is_system_admin()` mirrors `current_user_org()` — both are SECURITY DEFINER bypasses so RLS on dependent tables (`org_types` today; likely more later) doesn't have to reason about the admin table's own policies. Empty-by-default with no INSERT policy means no user can self-elevate through the app; bootstrapping the first sysadmin is always a conscious operator action.
**Alternatives rejected**:
- `is_system_admin boolean` on `"Person(s)"`: couples a cross-tenant role to tenant-scoped rows; a sysadmin would need a Person row in some arbitrary org, which misrepresents the scope.
- Supabase custom JWT claim / service_role dependency: admin identity would live outside the schema (in auth config), breaking grep-ability and making audits harder. Also splits the "who is an admin" source of truth across two systems.
- Allowing `authenticated` users to INSERT into `system_admins` (admins-grant-admins): would make elevation a runtime concern reachable through the normal auth path; the current model forces out-of-band action.

## 2026-04-22 — Solo-Entity per Person; Person(s) gets first/last/username

**Affects**: Person(s), Entity(s), Entity(s)_members, House Rule 8, House Rule 9
**Decision**: Every `"Person(s)"` insert auto-creates a 1-member `"Entity(s)"` row (`solo_for_person_id = person.id`, `name = person.username`) plus an `"Entity(s)_members"` row binding them. Name syncs on first/last name update via `AFTER UPDATE OF first_name, last_name` trigger. Cascades on Person delete via FK. `"Person(s)"` gains `first_name text NOT NULL`, `last_name text NOT NULL`, `username text GENERATED ALWAYS AS (first_name || ' ' || last_name) STORED` with NO uniqueness constraint, plus `is_active`, `deactivated_at`, `deleted_at`, `updated_at`, `created_by_person_id`, `updated_by_person_id`. New RLS helper `person_solo_entity_id(uuid)` returns the solo Entity for a given Person.
**Why**: Every downstream "who" column (role junctions on moment nodes, attest generator, policy author, etc.) needed a uniform FK target. Without solo-Entity, each such column had to be either polymorphic (`person_id XOR entity_id` discriminated union, fanning out across ~8 tables) or force users to manually wrap each Person in an Entity before they could be referenced. The solo-Entity pattern keeps the FK type singular (`entity_id → "Entity(s)".id`) everywhere while the trigger handles the wrapping invisibly. Username collisions allowed because real human names collide; failing an insert because a John Smith already exists is bad UX.
**Alternatives rejected**:
- Polymorphic `person_id XOR entity_id` on every "who" column: 2 FK columns × 8+ call sites = constant surface area cost for zero semantic gain once solo-Entities exist anyway.
- Unified parties table with role discriminator: loses per-role index locality and per-role RLS tuning.
- Unique `username` per org: blocks legitimate duplicate names like two "John Smith"s in the same org.
- `username` as a regular text column kept in sync by trigger: GENERATED STORED is atomic with the insert — no sync-lag window, no "trigger forgot to fire" failure mode.

## 2026-04-22 — Workflow / Process / Node: Template + Instance split, copy-on-spawn

**Affects**: workflow_template(s), workflow(s), process_template(s), process(s), node_template(s), moment_node(s), workflow_template_processes, House Rule 13
**Decision**: Three pairs of template + instance tables. Templates compose many-to-many at their level (`workflow_template_processes` junction between workflow and process templates). Instances are owned one-to-many (`process(s).workflow_id` FK, not a junction). Node templates live in a separate table from moment nodes; at process spawn, non-conditional node templates are copied eagerly into `"moment_node(s)"` with `spawned_from_template_id` set. Conditional node templates are spawned lazily — no row until the trigger+condition actually fires. Template edits do NOT retroactively propagate to already-spawned instances.
**Why**: Templates and instances have different lifecycles (templates edited rarely, cloned many times; instances created constantly, edited during their run) and different permissions (authoring a template is usually a narrower role than spawning one). Copy-on-spawn preserves historical integrity — editing the "Onboarding" process template shouldn't rewrite last quarter's completed onboardings. M:N composition at the template layer lets one process template be reused across workflow templates (e.g. "Approval" used by both "New Hire" and "Contractor Onboarding"); 1:M at the instance layer reflects that a running process instance belongs to exactly one running workflow — sharing a live process between two workflows would be concurrency hell. Mixed eager/lazy spawn was chosen over eager-all to avoid storage overhead on conditional branches that may never fire; over lazy-all because "what's currently pending?" becomes a simple SELECT with eager non-conditionals.
**Alternatives rejected**:
- Single table with `is_template` boolean flag: risks leaking templates into operational queries (every SELECT needs `WHERE is_template = false`); conflates access patterns.
- Live FK reference (instance points at template): template edits would retroactively mutate past runs — unacceptable for audit.
- 1:1 template-to-workflow: forces duplicating process templates for each workflow that needs them, defeating the point of templates.
- Fully eager spawn: stores rows for conditional branches that may never fire; multiplies partition row count.
- Fully lazy spawn: every "what's next?" query becomes a recursive walk of the template DAG minus what's already materialized — more app-layer complexity.

## 2026-04-22 — Moment nodes: one LIST-partitioned table, not four

**Affects**: moment_node(s), moment_node_sources, moment_node_owners, moment_node_affected, decision_justifications, task_prescription_options, attest(s), House Rule 11
**Decision**: Event / Action / Decision / Task all live in a single `"moment_node(s)"` table, LIST-partitioned on `node_type` with four child partitions. Primary key is composite `(id, node_type)` — Postgres requires the partition key in any unique constraint on a partitioned table. Every table that references a moment node therefore carries both `*_node_id` and `*_node_type` columns, and the FK is composite: `FOREIGN KEY (node_id, node_type) REFERENCES "moment_node(s)"(id, node_type)`. Scale target: tens of millions of rows per tenant.
**Why**: The four types are semantically distinct but constantly inter-refer — Trigger chains cross types (any node can trigger any other), role junctions (source/owner/affected) apply identically to all four, and cross-type queries like "everything Peter touched this week" are routine. Four separate tables would force discriminated FKs everywhere (`trigger_event_id` / `trigger_action_id` / `trigger_decision_id` / `trigger_task_id`) and UNION every cross-type query. LIST partitioning gives most of the physical-separation benefits of four tables — separate indexes, separate VACUUM, query-pruning — with none of the cross-reference pain. The composite-FK tax is real but predictable: the partition key companion column is a useful query filter in its own right.
**Alternatives rejected**:
- Four separate tables: ugly discriminated FKs, UNION-heavy queries, schema changes required on every referring table when a 5th node type is added.
- Single unpartitioned table: would handle 100M rows but not the user-stated "tens of millions per org × many orgs" ceiling; VACUUM and index bloat would dominate at scale.
- HASH sub-partition by `org_id`: premature; LIST-by-type alone is sufficient until cross-tenant scans dominate query time.
- RANGE sub-partition by month of `trigger_timestamp`: premature until archival patterns are clear.

## 2026-04-22 — Moment-node roles: three Entity-only junction tables

**Affects**: moment_node_sources, moment_node_owners, moment_node_affected, House Rule 8
**Decision**: The three A.1 roles (Originating Source, Owning Entity, Affected Entity) each get a dedicated junction table with identical shape: `(moment_node_id, moment_node_type, entity_id)`. All three target `"Entity(s)"` only — Person references go through the auto-created solo Entity per Rule 8. Non-empty invariant (≥1 row per role per node) is enforced at the app layer for now; no deferred DB constraint trigger yet. "Accountable Entity" was renamed to "Affected Entity" and made multi-valued like the other two.
**Why**: All three roles share cardinality (multi, ≥1) and target type (Entity). Three tables with identical shape is cleaner than one unified `moment_node_parties` table with a role discriminator — separate indexes per role, per-role RLS tuning possible, and "show me owners of this node" is a single-table scan instead of a filtered scan of a 3x-larger table. Entity-only targets (leveraging solo-Entity) eliminate discriminated-union FKs on every junction row. Renaming Accountable → Affected and making it multi-valued was a deliberate user choice: Accountable implies singular accountability (RACI's "one throat to choke"), Affected captures impact without demanding a designated owner, and uniformity of shape across roles was valued over the RACI convention.
**Alternatives rejected**:
- One unified `moment_node_parties` table with a `role text` column: role filter on every query; harder to tune indexes per-role.
- Discriminated `(person_id XOR entity_id)` on each junction row: redundant with solo-Entity pattern; every query would need a UNION-like resolution to unify the two sides.
- Accountable as a singular FK (RACI-style): user explicitly chose uniformity across roles over RACI orthodoxy.

## 2026-04-22 — Trigger on moment_node is single, nullable, node-only

**Affects**: moment_node(s).trigger_node_id, moment_node(s).trigger_node_type
**Decision**: A moment node's trigger is at most one other moment node (nullable single FK, not a junction). Trigger does NOT point directly at a Process — the process chain is reached transitively via the triggering node's own `process_id`. When `trigger_node_id IS NOT NULL`, this node's Originating Source is conceptually inherited from the trigger node's Source (derived via chain walk at read time; not denormalized). The reverse relation ("Cascade" — nodes that cite this one as their trigger) is derived, not stored.
**Why**: Real-world causality at the node level is almost always single-upstream. Multi-trigger breaks the "inherit source from trigger" rule (union of sources? pick one?). Keeping it single keeps the graph a DAG rather than a mesh, and keeps source derivation deterministic. Letting Trigger also target a Process would create two paths to the process from a node (the direct `process_id` and the trigger), inviting inconsistency; Process-level triggering is instead captured by `process(s).generated_by_node_id` on the process row itself. Deriving rather than denormalizing source preserves single-source-of-truth.
**Alternatives rejected**:
- Multi-trigger junction: source-inheritance rule becomes ambiguous with multiple upstreams.
- Trigger points at Node OR Process (discriminated): two paths to reach the process is confusion, not flexibility.
- Denormalized source copy on trigger-set: staleness risk when upstream source changes; solves a perf problem that doesn't exist yet.

## 2026-04-22 — Policy nodes live in their own table, not as a moment_node type

**Affects**: policy(s), policy_supporting_evidence, obligation_kind, task_prescription_options, House Rule 15
**Decision**: `"policy(s)"` is a dedicated table for authored rules. Columns include `kind obligation_kind` (ENUM: `recommended | mandatory | prohibitive`) and `author_entity_id uuid` FK to `"Entity(s)"` — individuals OR groups can author policies. Supporting evidence is a polymorphic junction (`policy_supporting_evidence`) whose `target_kind` is either `evidence_node` or `evidence_property`. The `obligation_kind` ENUM is shared with `task_prescription_options.obligation`.
**Why**: Policies are authored once and referenced many times; moment nodes are time-stamped events. Different lifecycles, different access patterns, different audit semantics. Mixing them would mean a Decision's "justification FK" could point at a sibling moment node instead of a policy — category error. Author as Entity (not Person) allows committee- or department-authored policies, which are legitimate and common in real org governance. Shared ENUM (rather than duplicated CHECK constraints) prevents drift when the list of obligation kinds ever changes.
**Alternatives rejected**:
- Policy as a 5th `node_type` on `moment_node(s)`: category error; policies aren't time-stamped events and don't participate in Trigger chains.
- Author as Person only: blocks committee / department authorship.
- Per-table CHECK constraints `IN ('recommended','mandatory','prohibitive')`: drift risk when values evolve.

## 2026-04-22 — Evidence as node + property, property rows append-only with supersession

**Affects**: evidence_node(s), evidence_property(s), House Rule 14
**Decision**: Evidence splits into `"evidence_node(s)"` (the citation: document / database row / trigger output, with `source_type` and `validity_kind`) and `"evidence_property(s)"` (specific claims extracted from a node; append-only). Evidence nodes are editable in place (metadata describes the citation, not the facts). Evidence properties are immutable: a BEFORE UPDATE trigger rejects any change to value, name, or captured_at columns. New values become new rows; the old row's `superseded_by_id` points at the replacement. Queries for "current" values filter `WHERE superseded_by_id IS NULL`.
**Why**: Historical decisions must cite the claim as it was at the time of reference. If a phone-number claim silently changes after a Decision was justified by it, the justification becomes a lie — the audit trail is broken. Supersession chain preserves cite-as-was while making the old row visibly outdated. Node-level metadata (name, storage_uri, description) doesn't need this treatment because it describes the REFERENCE, not the data.
**Alternatives rejected**:
- In-place edit with change_log audit table: Decision citations drift silently as values update; historical integrity is only recoverable by cross-referencing the log.
- JSONB array of properties on evidence_node: no FK integrity, no per-property supersession, no way to attest on individual claims.
- Append-only on both node AND property: the node is a reference, not a claim; metadata updates (fixing a typo in the document name) are fine and common.

## 2026-04-22 — Attests: polymorphic endorsements with target-branched RLS

**Affects**: attest(s), attest_comments, attest_visibility, Person(s) cross-org readability, House Rule 12
**Decision**: `"attest(s)"` is a single polymorphic endorsement table covering likes, approvals, vouches, agreements, and formal sign-offs. `generator_person_id` is a Person, not Entity — individual accountability is a schema invariant. Target is polymorphic (Rule 12): `target_kind text` ∈ `person | moment_node | process | policy | evidence_node | evidence_property`, `target_id uuid`, `target_node_type text` when kind is moment_node. Scalar fields: `polarity ∈ positive | negative | neutral`, `basis ∈ fact | opinion`, nullable `comment`, optional `process_id` context, optional `attest_start` / `attest_end` validity window, nullable `revoked_at`. **RLS branches on target_kind**: `target_kind = 'person'` attests are cross-org visible (for "vouch for a real person / phone / email" workflows); all other targets are org-scoped. `attest_visibility` ACL (person / entity / org rows) can widen visibility further, including cross-org grants.
**Why**: The data shape is identical whether it's a thumbs-up on a photo or a CEO sign-off on a billion-dollar decision — (who, kind, target, when, validity, revocation) applies to both. App logic handles the different business rules per kind. Person-as-generator enforces individual accountability: a committee approval is modeled as N individual attests, preserving who actually voted yes. Target-branched RLS is the cross-org corroboration primitive: users across tenants can attest to facts about a shared Person (confirming identity, phone number, email), but attests on tenant-internal things (moment nodes, processes) stay tenant-private. This is the first deliberate cross-tenant read path in the schema outside `"Entity(s)_members"`.
**Alternatives rejected**:
- Split into `endorsements` (heavyweight, append-only, time-bounded) vs `reactions` (lightweight, toggleable): same underlying shape, app-layer distinction is cheaper than two tables + unified-view plumbing.
- Entity as generator: loses individual accountability; committee votes become opaque aggregates.
- All attests org-scoped: blocks the cross-org Person-corroboration use case entirely.
- `affirmative` rather than `attest`: naming collision — "affirmative" connotes positive, but the field must also hold dislikes (negative polarity). User renamed to `attest` and added explicit `polarity` column.

## 2026-04-22 — Polymorphic target refs: target_kind + target_id (+ companion when partitioned)

**Affects**: attest(s), decision_justifications, policy_supporting_evidence, attest_visibility, status_change_log, House Rule 12
**Decision**: Tables that hold heterogeneous references use a thinner pattern than Rule 3 fan-out: `target_kind text` (with CHECK list of allowed kinds) + `target_id uuid`. When `target_kind` resolves to a partitioned table (currently only `moment_node`), a companion `target_node_type` column is populated to satisfy the composite-FK requirement of Rule 11. FK integrity is application-layer, not database — no single FK can point at multiple parent tables. Violations surface via test coverage and occasional SQL audits.
**Why**: Five+ tables need this same shape. Fan-out with per-type FK columns (Rule 3) is correct for `"Entity(s)_members"` because the 7-way member type set is small, stable, and tenant-critical. For attests, decision justifications, and the audit log, fan-out would add 6 nullable FKs to each table — a lot of columns to index and constrain when the app already naturally filters by `target_kind`. Accepting app-layer FK integrity here is a pragmatic trade-off for schema width.
**Alternatives rejected**:
- Rule-3 fan-out on all five tables: 30+ nullable FK columns across the affected tables; schema-wide pressure from each new target kind added.
- A central `targets` table with surrogate IDs: every target resolution becomes a 2-hop join; doesn't actually gain FK integrity for polymorphic refs (you'd still need a discriminator inside `targets`).

## 2026-04-22 — Milestones use flat AND/OR (DNF-lite)

**Affects**: process_template_milestones, milestone_conditions, process_milestone_hits
**Decision**: Each milestone has a single `logic_op text ∈ ('AND','OR')`. All of its `milestone_conditions` rows (each naming a `node_template_id` + `required_status`) are combined by that one operator. Nested Boolean logic like `(A AND B) OR (C AND D)` is NOT supported at this layer. Milestone hits are logged per process instance in `process_milestone_hits` with `UNIQUE (process_id, milestone_id)` — a milestone fires at most once per instance.
**Why**: ~90% of real-world milestones are flat AND or flat OR. Encoding nested Boolean logic as a table structure adds more machinery than it earns. The upgrade path is non-destructive: add a `milestone_condition_groups` layer between milestone and conditions; existing flat-AND milestones become "one group with N conditions"; flat-OR becomes "N groups with one condition each"; full DNF becomes possible. Can be done without migrating existing rows.
**Alternatives rejected**:
- Full nested Boolean tree stored as JSONB: no FK integrity on embedded node_template_id refs.
- Deeply normalized Boolean tree (recursive groups): complex at a scale nobody has asked for.
- Milestones that can fire multiple times per instance: loses the "milestone hit" semantic — a milestone is a one-shot event per run.

## 2026-04-22 — Task prescription options are a side table, not per-option Task nodes

**Affects**: task_prescription_options
**Decision**: A Task node's "prescription option set" lives in a dedicated `task_prescription_options` side table — one row per option, with `obligation obligation_kind`, `is_conditional`, `condition_text`, optional `policy_id` backing, and `position`. Each option is a CANDIDATE action, not itself a Task node. When someone actually acts on (or explicitly declines) an option, that's an Action node, linked via the Trigger chain.
**Why**: The quality-analysis use case depends on being able to reason about options that were *available* but *not chosen* — the user's "Peter and Joe take no action and allow Alex to continue" example requires explicit representation of what could have happened. Per-option-is-Task collapses that distinction (if the Task doesn't exist, there's nothing to analyze). Side table also lets each option FK to its backing Policy cleanly and supports attests on individual options, not just whole Tasks. Revisited mid-conversation — initial leaning was per-option-is-Task (emergent via Trigger chain), changed to side table after reviewing the user's original "Prescription Option Set: Action(s) each labeled..." phrasing.
**Alternatives rejected**:
- Each option is its own Task node (emergent via Trigger): can't represent "option existed but wasn't chosen"; loses set-level metadata like "exactly one must be chosen" vs "pick any combination."
- JSONB column on `moment_node(s).prescription_options`: no FK to Policy for backing; no per-option attest targets; no queryability for "which tasks have a Prohibitive option?" across millions of rows.

## 2026-04-22 — Intent docs live in this repo, split across four files

**Affects**: Repo structure, CLAUDE.md sync rule, SCHEMA.md, DECISIONS.md, UPDATING.md
**Decision**: Schema intent lives in a dedicated repo (`Schema`), separate from any app code. Four files: `CLAUDE.md` (session rules), `SCHEMA.md` (current-state, always true), `DECISIONS.md` (append-only log), `UPDATING.md` (editing protocol). - Full ADR per decision (one file each in `decisions/`): overhead without payoff at single-user scale.
- Auto-generated markdown from `pg_catalog`: shifts fragility from doc to generator.
- Hooks on `apply_migration` (Q4 option C): premature; `CLAUDE.md` rule is cheaper and sufficient until drift is actually observed.
- Co-locating in the existing `Data Storage` repo (Schema Planner): semantic mismatch — Schema Planner's CLAUDE.md documents itself, not a tenant app.

## 2026-04-22 — Sharing delegation is a separate table, not a flag on shares

**Affects**: Person(s)_contact_share_delegations, Person(s)_contact_shares
**Decision**: View access and re-share authority are distinct tables — `Person(s)_contact_shares` (view grants) and `Person(s)_contact_share_delegations` (re-share authority). A delegate does not need a share row to have authority; they already have view access via same-org RLS.
**Why**: Reading from the owner's own org is the common case for delegates. A `can_reshare` flag on shares would force inserting "share with my own coworker" rows purely to mark re-share authority — semantic noise. Separate tables also give cleaner revocation semantics: revoking a delegation cascades to shares-by-delegate; revoking a direct share touches only that row.
**Alternatives rejected**:
- `can_reshare` boolean on shares: creates the semantic-noise problem above.
- Implicit delegation (anyone in owner's org can re-share by default): wrong default for privacy; user wanted explicit opt-in.

## 2026-04-22 — Share target is polymorphic over (user, entity) only

**Affects**: Person(s)_contact_shares, Entity(s)
**Decision**: Share target is `target_user_id uuid` XOR `target_entity_id uuid`. No direct FK columns for org or Div1..5. Sharing with a whole org or a div means creating (or reusing) an `Entity(s)` that wraps it, then sharing with the Entity.
**Why**: `Entity(s)_members` is already polymorphic across Person/Org/Div1..5. Duplicating that polymorphism at the share level would mean 7 FK columns here and 7 there — redundant, and every audit query unions across both anyway. Users remain a direct target because single-user shares are common enough that forcing Entity materialization would be UX friction.
**Alternatives rejected**:
- Option 1 (fat polymorphism, 7 FK columns including `target_org_id`, `target_div1_id`..`target_div5_id`): redundant with `Entity(s)_members`.
- Option 3 (entity-only, including single-user shares): forces materializing an Entity for each one-off share — UX friction without structural payoff.

## 2026-04-22 — Visibility tiers: three-level enum per contact row, default `'org'`

**Affects**: Person(s)_emails, Person(s)_phones, can_view_person_contact()
**Decision**: Each contact row has `visibility text CHECK IN ('private','org','shareable')` with default `'org'`. Sharing is opt-in per row — a row must be `'shareable'` to be visible to share targets. RLS SELECT uses `can_view_person_contact(owner, visibility, channel)` which returns true for (self) OR (same-org AND tier ∈ {org, shareable}) OR (active share AND tier = shareable).
**Why**: Three tiers exactly match the three relationship types that can view a Person's contact (self, same-org, shared). Per-row granularity lets a Person keep some emails (e.g. "spam") private while marking "work" as shareable. Default `'org'` means new rows are internally visible but not externally shareable — the safer default.
**Alternatives rejected**:
- Per-row explicit grant table (no tiers): max flexibility, excessive bookkeeping — every share would require enumerating rows.
- Tier + per-row overrides (hybrid): more machinery than needed.
- `'shareable'` as the default: optimistic default inverts privacy ergonomics.

## 2026-04-22 — Contact rows are "2 active + archived history", not "3 equal slots"

**Affects**: Person(s)_emails, Person(s)_phones
**Decision**: UI shows 2 active emails/phones; the 3rd slot is a history view of archived rows. DB enforces max 2 active via `UNIQUE (person_id, ordinal) WHERE status='active'`. Content changes auto-archive the OLD snapshot via a BEFORE UPDATE trigger (`Person(s)_emails_archive_on_change`, `Person(s)_phones_archive_on_change`).
**Why**: The user's actual requirement was "2 current + a history channel", not "3 equivalent slots". Trigger-driven archival means even a buggy UPDATE preserves the prior value — app-layer archival could be skipped, causing silent data loss.
**Alternatives rejected**:
- 3 equal current slots: wrong model for the UI described.
- App-layer archival: trust-based, can be skipped.
- Separate `_history` table: extra join for the common "show history" query; no gain over `status` column.

## 2026-04-22 — Entity(s): tenant-scoped, non-nested, name-unique per org

**Affects**: Entity(s), Entity(s)_members
**Decision**: `"Entity(s)"` table has `UNIQUE (org_id, name)` — duplicate entity names allowed across orgs, blocked within one. No nesting: an Entity cannot have another Entity as a member. Cross-tenant membership IS allowed (member's own org need not match the entity's org) — intentional, gated at the RLS layer, not by DB constraint.
**Why**: Two orgs may each legitimately have an entity called "Leadership"; that should not collide. Within a single org, two "Leadership" entities would be a definition conflict. Nesting was declined to avoid recursion in membership-expansion semantics. Cross-tenant membership is required for the contact-sharing use case (sharing with a Person in another org).
**Alternatives rejected**:
- Global-unique entity name: false collisions across tenants.
- No unique constraint at all: data drift risk.
- Nested entities: ambiguous member-expansion (does a nested-member's membership transit through?) not worth the complexity.
- DB-enforced tenant boundary on members (trigger checking `entity.org_id = member.org_id`): would block the legitimate cross-tenant use case.

## 2026-04-22 — `member_*` prefix on Entity(s)_members; Entity(s) gets full CRUD RLS

**Affects**: Entity(s), Entity(s)_members, RLS policies
**Decision**: Member FK columns in `Entity(s)_members` are named `member_person_id`, `member_org_id`, `member_div1_id`..`member_div5_id` — not `org_id`. `"Entity(s)"` and `"Entity(s)_members"` get full CRUD RLS policies (SELECT/INSERT/UPDATE/DELETE) gated on `org_id = current_user_org()`, deviating from the existing `Div1`/`Person(s)` pattern of SELECT-only with writes going through service role.
**Why**: `org_id` already means "tenant" elsewhere in this schema — reusing it for "which org is this member referring to" would be a readability trap. Prefixing makes role explicit. Full CRUD on Entity(s) was chosen because entities are user-created objects, unlike Persons/Divs which are admin-managed — end-users need to create and edit their own entities from the app.
**Alternatives rejected**:
- Reuse `org_id` column name: ambiguity between tenant-role and member-role.
- Read-only + service_role writes (existing pattern): doesn't fit the user-created-object model.
